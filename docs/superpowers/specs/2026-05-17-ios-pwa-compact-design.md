# iOS PWA (compact scope) — Design

**Date:** 2026-05-17
**Status:** Approved (design phase)
**Scope:** Make the existing `mobile/www/` web bundle installable on iOS as a
fullscreen standalone app via Safari "Add to Home Screen." Installable shell
only — no offline, no service worker, no native wrapper.

## Goal

iOS users open the bui URL in Safari → "Add to Home Screen" → launch a
fullscreen standalone app with the bui icon. The app reaches the backend
through the existing cloudflare quick-tunnel pattern already documented in
AGENTS.md. No backend or app-logic changes.

## Why this is small

The mobile app is already a static web bundle (`mobile/www/`: `index.html` +
one JS + one CSS) that talks to the Linux box `src/server/` over HTTP+WS/SSE
via the `window.api` shim (`src/renderer/api/httpApi.ts`). Capacitor is only a
native wrapper around that bundle; the config uses **no native plugins**. A
PWA is therefore purely additive static metadata on the already-served bundle.

## Architecture

```
iPhone Safari → https://<random>.trycloudflare.com   (existing cloudflared
                  ↓ serves                              quick-tunnel)
                mobile/www/index.html                  (existing src/server/
                  + <link rel="manifest">       ← NEW    static serve)
                  + apple-mobile-web-app-* meta  ← NEW
                  + apple-touch-icon link        ← NEW
                manifest.webmanifest             ← NEW (source-controlled)
                icons/icon-180.png, icon-512.png ← NEW (derived from
                                                        mobile/assets/icon.png)
```

No code path changes. The PWA is static metadata served alongside the
existing bundle.

## Components

### 1. `manifest.webmanifest` (new, source-controlled)
Minimal manifest:
- `name`: "Better UI"
- `short_name`: "bui"
- `display`: "standalone"
- `background_color` / `theme_color`: "#0e0f12" (app dark bg)
- `start_url`: "." (relative — works under any tunnel hostname)
- `icons`: 180 (any) + 512 (any + maskable)

### 2. App icons (new)
Derived from existing `mobile/assets/icon.png` (1024×1024) using `sips`
(macOS built-in, zero dependency):
- `icon-180.png` — `apple-touch-icon`, iOS home screen
- `icon-512.png` — manifest icon

### 3. `index.html` tag injection (the one nuance)
`mobile/www/index.html` is **generated by `npm run build:mobile`** (Vite via
`electron.vite.config.mobile.ts`). Editing the emitted file directly is wiped
on rebuild. The **source HTML template** Vite consumes must carry the tags so
they survive rebuilds. Tags to add:
- `<link rel="manifest" href="./manifest.webmanifest">`
- `<meta name="apple-mobile-web-app-capable" content="yes">`
- `<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">`
- `<meta name="apple-mobile-web-app-title" content="bui">`
- `<link rel="apple-touch-icon" href="./icons/icon-180.png">`

The manifest + icons are static assets that must land in `mobile/www/` on
build. They go in the mobile build's Vite `publicDir` (exact path confirmed
during planning) so `build:mobile` emits them automatically and they stay in
sync — never hand-copied into the generated output.

### 4. CSP
`index.html` carries a `Content-Security-Policy` meta with
`default-src 'self'`, which covers a same-origin manifest. No CSP change
expected; verified during testing (manifest must load with no CSP violation).

## Data flow / lifecycle

1. Linux box: `BUI_MOBILE_HOST=127.0.0.1 npm run mobile` +
   `cloudflared tunnel --url http://127.0.0.1:8787 --protocol http2`
   → `https://<random>.trycloudflare.com`.
2. Open in iOS Safari → Safari reads manifest + apple meta tags.
3. Share → Add to Home Screen → installs with bui icon,
   `apple-mobile-web-app-title` as label.
4. Launch → opens `start_url: "."` (the tunnel origin it was installed from)
   in **standalone mode**: no Safari chrome, fullscreen, existing `.mobile`
   shell renders as today.
5. Backend reached over the same origin via the existing `window.api`
   HTTP+SSE shim. No code path changes.

**Ephemeral-URL behavior (accepted, scope A):** the installed icon is pinned
to the tunnel hostname it was added from. New tunnel session → new hostname →
old icon is dead; re-add to home screen. Known and accepted for this scope.

## Error handling / edge cases

- **`black-translucent` status bar:** content draws under the iOS status bar.
  The mobile shell already applies `env(safe-area-inset-*)` to `.mobile-body`
  (prior work), so the top inset is covered. Verify the header isn't clipped
  under the notch during testing; if it is, fix via the same `.mobile`-scoped
  safe-area pattern — not a manifest change.
- **No service worker:** zero offline. Opening the icon offline shows Safari's
  standard failure — acceptable at scope A.
- **Manifest ignored by iOS:** the apple-`meta` tags alone still yield a
  usable fullscreen add-to-home-screen (iOS historically relies more on the
  apple meta tags than the manifest for standalone) — belt-and-suspenders.

## Testing / verification

1. `npm run build:mobile` → confirm `manifest.webmanifest` + `icons/` land in
   `mobile/www/` and the emitted `index.html` contains the injected tags
   (proves the build doesn't wipe them).
2. Serve `mobile/www/` locally, load at mobile viewport, confirm: manifest
   parses (no console/CSP error), icons resolve 200, `display: standalone`.
3. **On-device (iOS):** the real proof is Add-to-Home-Screen on an actual
   iPhone via the cloudflare tunnel, launching fullscreen with the bui icon.
   Cannot be driven from here — verify everything locally + from build output,
   then provide a precise manual on-device checklist (same "device-verified"
   honesty as the Android work).

## Desktop / Android non-regression

All changes are confined to the mobile build (`mobile/www/` assets + the
mobile Vite HTML template). Desktop Electron build and the Capacitor Android
app are untouched. The injected apple/manifest tags are inert on desktop
(Electron loads its own `index.html`).

## Out of scope (explicit)

- Service worker / offline caching
- Named / stable cloudflare tunnel (ephemeral URL accepted)
- Splash screens, custom install-prompt UX
- iOS standalone navigation quirk handling beyond status-bar inset
- Any Android / Capacitor / desktop change
